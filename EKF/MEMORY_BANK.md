## 3. 原因分析

### ふらふらの主原因（推定）

981ee89 での位置制御不安定は、以下の **4つのパラメータ設定ミス** が複合的に作用した結果と考えられる。

#### 🔴 原因1: `EK3_GPS_CHECK = 0`（GPS健全性チェック無効）

- **影響**: GPSが異常値を出力しても、EKFがそれをそのまま取り込んでしまう
- **結果**: 突発的なGPSノイズや異常値が位置推定に直接反映され、機体がふらつく
- **修正**: `1` に設定し、健全性チェックを有効化

#### 🔴 原因2: `EK3_GLITCH_RAD = 0`（GPSグリッチ検出無効）

- **影響**: GPSの突然のジャンプ（グリッチ）が検出されず、位置推定が大きく乱れる
- **結果**: グリッチ発生時にEKFが即座にそれを追従しようとし、急激な位置補正が発生
- **修正**: `5`m に設定し、5m以上のGPSジャンプをグリッチとして検出・排除

#### 🔴 原因3: ノイズパラメータの過小設定（`EK3_POSNE_M_NSE = 0.1`, `EK3_VELNE_M_NSE = 0.15`）

- **影響**: EKFがGPS測定値を**過度に信用**しすぎる
- **背景**: MAVLink GPS Injection（`GPS_TYPE = 14`）環境では、通常のGPSより測定値のノイズが大きい
- **結果**: ノイズの多いGPSデータにEKFが過剰に追従し、位置推定が振動的になる
- **修正**: ノイズ値を標準の約**2倍**に設定（`POSNE: 0.2`, `VELNE: 0.3`, `VELD: 0.5`）

#### 🟡 原因4: `EK3_CHECK_SCALE = 200`（チェックが寛容すぎる）

- **影響**: 異常な測定値を通常の200%まで許容してしまう
- **結果**: 上記のGPS異常値がチェックを通過しやすくなり、問題を悪化させる
- **修正**: `100`%（標準値）に戻し、正常範囲の測定値のみを受け入れる

### 相互作用のメカニズム

```
GPS異常値発生
    ↓
EK3_GPS_CHECK=0 → 健全性チェックをスキップ
    ↓
EK3_GLITCH_RAD=0 → グリッチ検出されず
    ↓
EK3_CHECK_SCALE=200 → 異常値でも寛容に許可
    ↓
EK3_POSNE_M_NSE=0.1 → GPSを信用しすぎて即追従
    ↓
位置推定が大きく乱れる → ふらふら
```

## 4. 結論・教訓

### Observer4 パラメータ変更時の鉄則

| # | ルール | 理由 |
|---|--------|------|
| 1 | **`EK3_GPS_CHECK` は必ず `1`（有効）** | GPS健全性チェックを無効にすると、異常値がEKFに直接流入し位置制御が破綻する |
| 2 | **`EK3_GLITCH_RAD` は `0` にしない（最低 `5` 推奨）** | GPSグリッチ検出は位置制御の最後の砦。無効化すると突然のGPSジャンプに無防備になる |
| 3 | **MAVLink GPS Injection 環境ではノイズ値を標準の約2倍に設定** | `GPS_TYPE = 14`（MAVLink Injection）は通常GPSよりノイズが大きい。`EK3_POSNE_M_NSE`, `EK3_VELNE_M_NSE`, `EK3_VELD_M_NSE` を標準の2倍程度にして、EKFがGPSを過信しないようにする |
| 4 | **`EK3_CHECK_SCALE` は `100` が安全** | `200` にすると異常値が通過しやすくなり、上記の問題が増幅される。特別な理由がない限りデフォルト値 `100` を維持する |

### 安定版パラメータセット（775a238）の要点

```
# GPS保護（最重要）
EK3_GPS_CHECK   = 1     # 健全性チェック有効
EK3_GLITCH_RAD  = 5     # 5m以上のGPSジャンプを検出
EK3_CHECK_SCALE = 100   # 標準的なチェック厳格度

# ノイズ設定（MAVLink Injection用、標準の約2倍）
EK3_POSNE_M_NSE = 0.2   # 水平位置ノイズ
EK3_VELNE_M_NSE = 0.3   # 水平速度ノイズ
EK3_VELD_M_NSE  = 0.5   # 垂直速度ノイズ
```
